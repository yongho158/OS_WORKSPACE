운영체제 Term Project 구현 체크리스트
====================================

목표
----
- 공항 체크인 카운터 5개를 대상으로 비선점형 멀티레벨 스케줄링 시뮬레이터를 구현한다.
- 기본 데이터셋(input.txt)으로 직접 설계한 스케줄러와 Baseline 3종의 ATT를 비교한다.
- 최종보고서에 넣을 승객별 결과, 등급별 평균 Turnaround Time, 카운터별 통계, ATT 비교표/그래프를 생성한다.

4. 시뮬레이션 엔진 구현
----------------------
필수 조건:

- 이산 사건 시뮬레이션 방식으로 구현
- time unit 기준으로 상태 변화 기록
- 승객이 도착하면 ready queue에 추가
- 카운터가 비면 스케줄러가 승객 선택
- 승객이 서비스를 시작하면 완료될 때까지 중단하지 않음
- Context switching overhead는 0으로 처리
- 모든 카운터 처리 속도는 동일하게 처리

시뮬레이션 흐름:

1. 현재 시간 time = 0으로 시작
2. time에 도착한 승객들을 ready queue에 추가
3. 완료 시간이 된 카운터의 승객을 completion 처리
4. 비어 있는 카운터마다 SchedulerStrategy.select_next() 호출
5. 선택된 승객의 service_start_time 기록
6. completion_time = service_start_time + service_time 계산
7. 다음 이벤트 시간으로 이동
   - 다음 승객 도착 시간
   - 다음 서비스 완료 시간
8. 모든 승객이 완료될 때까지 반복

로그에 남길 상태 변화:

- 승객 도착
- 카운터 배정
- 서비스 시작
- 서비스 완료
- 카운터 idle 발생


5. SchedulerStrategy 구조 구현
------------------------------
공통 인터페이스:

- select_next_passenger(ready_queue, counters, current_time, counter)
  - 입력:
    - 현재 대기 중인 승객 목록
    - 카운터 목록
    - 현재 시각
    - 배정 대상 카운터
  - 출력:
    - 다음에 처리할 Passenger 1명
    - 처리할 승객이 없으면 None

주의:

- Baseline과 우리 팀 스케줄러가 같은 SimulationEngine을 사용해야 한다.
- 스케줄러만 교체 가능한 Strategy Pattern 구조여야 한다.
- 스케줄러 안에서 승객 완료 시간을 직접 바꾸지 말고, 선택만 담당하게 한다.


6. Baseline A 구현: 단순 FCFS
----------------------------
정책:

- 모든 승객을 단일 ready queue로 관리
- 도착 시간이 빠른 승객부터 선택
- 도착 시간이 같으면 passenger_id가 작은 승객부터 선택
- 카운터는 빈 카운터 순서대로 배정

정렬 기준:

1. arrival_time 오름차순
2. passenger_id 오름차순

보고서에 쓸 분석 포인트:

- 단순하고 공정하지만 긴 service_time 승객이 앞에 있으면 Convoy Effect가 발생할 수 있다.


7. Baseline B 구현: 등급별 고정 우선순위
--------------------------------------
정책:

- First > Business > Economy 순서로 우선 처리
- 같은 등급 안에서는 FCFS 적용
- 모든 카운터는 빈 카운터가 되면 가장 높은 우선순위 등급의 승객을 선택

정렬 기준:

1. passenger_class 오름차순
   - 1 First
   - 2 Business
   - 3 Economy
2. arrival_time 오름차순
3. passenger_id 오름차순

보고서에 쓸 분석 포인트:

- First/Business 승객의 Turnaround Time은 줄어들 수 있다.
- Economy 승객은 오래 밀려 전체 ATT가 나빠질 수 있다.
- Aging이 없으면 Starvation 위험이 있다.


8. Baseline C 구현: Non-preemptive SJF
-------------------------------------
정책:

- 등급을 무시하고 service_time이 가장 짧은 승객부터 선택
- 같은 service_time이면 먼저 도착한 승객 선택
- 그래도 같으면 passenger_id가 작은 승객 선택

정렬 기준:

1. service_time 오름차순
2. arrival_time 오름차순
3. passenger_id 오름차순

보고서에 쓸 분석 포인트:

- 평균 Turnaround Time을 줄이는 데 강하다.
- service_time이 긴 First 승객이 뒤로 밀릴 수 있다.
- 공정성과 등급 우선 정책 측면에서는 약점이 있다.


9. 우리 팀 스케줄러 구현
-----------------------
권장 설계 이름:

- Hybrid Multi-Level Queue Scheduler
- 또는 Weighted HRRN + SJF Scheduler

반드시 포함할 알고리즘 3가지 이상:

- Multi-Level Queue
- Non-preemptive SJF
- Priority Scheduling
- HRRN 또는 Aging
- FCFS tie-break

권장 큐 구조:

- First Queue
- Business Queue
- Economy Queue

권장 카운터 정책:

- C1: First 우선
- C2: Business 우선
- C3: Economy 우선
- C4: Flex
- C5: Flex
- 전용 카운터가 자기 등급 승객 없이 idle이면 다른 등급 승객도 처리 허용

승객 선택 점수 예시:

- waiting_time = current_time - arrival_time
- response_ratio = (waiting_time + service_time) / service_time
- class_weight:
  - First = 1.5
  - Business = 1.1
  - Economy = 1.0
- score = response_ratio * class_weight
- score가 높은 승객 우선
- 동점이면 service_time 짧은 승객 우선
- 그래도 동점이면 arrival_time 빠른 승객 우선
- 그래도 동점이면 passenger_id 작은 승객 우선

설계 의도:

- SJF 성격으로 짧은 작업을 빠르게 처리해 전체 ATT를 낮춘다.
- Priority 가중치로 First/Business 승객이 지나치게 밀리지 않게 한다.
- HRRN/Aging으로 오래 기다린 Economy 승객도 eventually 선택되게 한다.
- FCFS tie-break로 동일 조건에서 공정성을 유지한다.


10. 카운터 배정 로직 구현
-----------------------
구현할 것:

- idle 상태인 카운터 찾기
- 카운터별 우선 처리 등급 적용
- Flex 카운터는 전체 ready queue에서 스케줄러 점수가 가장 높은 승객 선택
- 전용 카운터는 다음 순서로 선택
  1. 자기 전용 등급 승객이 있으면 그 등급 큐에서 선택
  2. 자기 전용 등급 승객이 없으면 전체 큐에서 선택
  3. 선택 가능한 승객이 없으면 idle 유지

주의:

- 전용 카운터의 타 등급 처리 허용 여부는 보고서에 명확히 설명해야 한다.
- ATT가 유일한 성능 지표이므로 idle 시간을 줄이는 정책이 유리하다.

- First 평균 Turnaround Time

- Economy 평균 Turnaround Time

전체 계산:

- ATT = 모든 승객 turnaround_time 합 / 50

카운터별 계산:

- 처리 승객 수
- 총 처리 시간
- idle time
- 처리한 승객 ID 목록

18. Trade-off 분석 준비
----------------------
반드시 언급할 것:

- ATT를 줄이기 위해 어떤 등급의 대기 시간이 늘어났는지
- First/Business/Economy 간 평균 Turnaround Time 차이
- SJF 성격이 강하면 긴 service_time 승객이 손해를 볼 수 있음
- Priority 성격이 강하면 Economy 승객이 손해를 볼 수 있음
- HRRN/Aging은 starvation을 줄이지만 순수 SJF보다 ATT가 조금 증가할 수 있음
- Flex 카운터를 적극 활용하면 idle time은 줄지만 전용 카운터의 의미가 약해질 수 있음
